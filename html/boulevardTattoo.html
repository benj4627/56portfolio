Ud fra UX-research og kravspecifikation fra kunden skaber vi en fuld grænseflade, hvis formål er at forstærke The Boulevard Tattoos online tilstedeværelse. Derudover lægger grænsefladen vægt på butikkens værdier og egenskaber. I sidste ende kan vi præsentere en funktionel, intuitiv og visuel tiltalende, brugercentreret grænseflade. Sitet er realiseret med HTML, CSS og vanilla JavaScript.

UI Design

Min gruppe og jeg har benyttet googles design sprint til at udforme de første skitser til produktet. Et design sprint fokuserer på et område om dagen, over 4 dage, hvor man til sidst har et færdigt mockup.

Skitserne er baseret på den information vi har fået om vores brugere, kunden og deres behov fra henholdsvis kvalitative brugerundersøgelser og indsamling af standarddata omkring området. De endelige iterationer af vores skitser, bliver herefter oversat til wireframes hvor vi udarbejder vores endelige layout for produktet. Her har vi fokus på genanvendelige komponenter med henblik på at udnytte atomic design.

Når vores wireframes er færdige, begynder vi på processen med high fidelity mockups. Vi implementerer her vores valgte farver, typografi og indhold, ud fra vores styleguide og indhold fra vores content sitemap. Med det, bliver det mere tydeligt hvilke elementer i grænsefladen som hænger sammen. I løbet af processen laver vi forskellige komponenter vi kan genbruge på tværs af siderne i vores grænseflade, hvilket effektivisere arbejdet

Vi sørger for at holde os til vores wireframe, men små justeringer er ofte nødvendige når man arbejder med indholdet. Der er der afvigelser fra wireframe til mockup. Vi bruger en hero section på mange af siderne for den visuelle vægt det skaber, og samtidig for at understrege for brugerne hvilke underside de befinder sig på. Derudover udnytter vi Jakob Nielsen 10 principper om usability igennem hele grænsefladen.

HTML

Vi opsætter kodemiljøet baseret på en github template med allerede eksisterende basal styling og boileplate HTML. Vi bruger korrekt semantisk HTML strukfur for at optimere teknisk SEO og webtilgængelighed. Derudoer udfylder vi både meta description, title, favicon og alt tekster til samtlige billeder, på alle underside, af samme grund som korrekt HTML struktur.

CSS

Udover vores eksisterende base styling, opsætter vi CSS variabler for at kunne sikre konsistent mellem vores mockup og produkt, og samtidig for at gøre det nem mere for os alle sammen at programmere med samme farver, spacing, border radius osv. VI sikrer på den måde stilmæssigt konsistens på tværs af undersider, og samtidig har vi nemt ved at rette f.eks. spacing eller border radius senere i processen hvis det er nødvendigt. Vi udnytter herudover få utility classes som b.la. sikrer at vores indhold holder sig inden for 1440p på desktop. Det gør vi for at undgå at sidens indhold på store viewports, bliver uoverskueligt at interagere med. Altså, vises siden på en skærm større end 1440px, vil indholdet forblive centreret.

Derudover genbruger vi forskellige classes igen, for at skabe konsistens og for at sikre at vores realiseret produkt lever op til vores principper fra atomic design, fra vores mockup. For at lave vores layout bruger vi primært display:grid, og det er for at sikre at vi kan placere vores elementer korrekt if.t. hinanden, og samtidig for at udnytte fleksibiliteten grid har, hvis der på sigt skulle skiftet indhold, eller mængden af indhold vokser.

Vi udnytter media queris for at optimere vores grænseflade til en mobil viewport. Igen bruger vi grid, men her er formålet at vise de fleste elementer i enten en elle to kolonner. Samtidig sørger vi for at have en width på 90% på vores containere, for at skabe margin fra indholdet ud til kanten af viewporten.

JavaScript

Vi bruger JavaScript til funktionalitet, animationer og til at fetche data fra vores WordPress API. Vi benytter funktionen toggleClassScroll til at animere elementer, så de slider op fra bunden, når brugeren scroller. EventListener registrerer scroll-positionen og aktiverer funktionen, når scroll grænsen er nået. Elementerne får CSS-klassen animationVisible, som ændrer opacity fra 0 til 1 og justerer deres position.

Funktionen til vores popup chat og vores hamburger menu, både på mobil og desktop fanger først de relevante HTML elementer med document.querySelector, og herefter tilføjes der en eventListener som lytter efter klik. Når elementer trykkes på, toggler CSS klasser, der ændrer opacity fra 0 til 1 og derfor kan de vises og lukkes ned igen efterfølgende.

Derudover benytter vi en funktion til at skifte indholdet på to af vores underside, når brugeren trykker på forskellige tabs i grænsefladen. Med et forEach loop tilføjer vi eventListeners til vores tabs, og herefter kan vi ved hjælp af tabContent arrayet, tildele det korrekte indhold til de korrekte tabs. Indholdet bliver hentet fra dets plads i arrayet og derfor kan vi matche indhold med tabs. Til sidst kalder vi klik metoden på tabsBtns[0] for at simulere et klik på den første tab, hvis plads i arrayet er 0, når siden loades. Derfor er den første tab altid aktiv når siden loader.

Vi bruger funktionen filterProductsByCategory til at tilføje filtreringsmuligheder. Først fanger vi data-category-id fra det klikkede element i HTML’en og gemmer det i en variabel. FilterProductsByCategory gennemgår så de relevante elementer og henter deres produktkategorier. Hvis kategoriens ID matcher det ID, som vores links i HTML’en har, eller er 0, som repræsenterer alle kategorier, får elementerne display: flex, mens de, der ikke matcher, får display: none. Derfor vises kun de valgte.

Funktionen fetchKunsterID viser hentede data i HTML'en. Vi udtrækker værdierne fra vores JSON-objekt og gemmer dem i variabler. Herefter bruger vi variablerne til at opdatere HTML-indholdet ved hjælp af innerHTML, med template literals og string interpolation. Til sidst kaldes fetchKunsterID med det ID, der matcher den valgte kunster. Denne proces gentages for hver kunstner, som er i grænsefladen.

BEGRÆNSINGER

Projektet var tidsbestemt, og derfor er der selvfølgelig punkter optimere yderligere. Loadingtiden kan nedbringes betydeligt ved at have mindre JavaScript, optimere filstørrelser og generelt komprimere, der hvor det er muligt. Var projektet noget som skulle leveres til kunden i virkeligheden, havde der forekommet diverse finjusteringer og optimeringer, således produktet lever op til den standard man forventer af en fuld grænseflade.
